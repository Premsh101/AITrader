AITrader – User Manual

Step-by-step guide: from training the AI models to running the system safely.

1. Current readiness — in simple words

✅ READY — the software. All the bugs that stopped trading are fixed: symbol formats, security, the trading loop, order prices, and the database setup. 37 automated tests pass.
❌ NOT READY — the AI models. The old model files were trained on different data shapes and never worked. They are parked in models/legacy/. You must train new models on Kaggle first (Step 3 below). Until then the app runs, the dashboard works, but it will refuse to trade — on purpose.
⚠️ NEEDS YOUR INPUT — configuration. Before deploying you must set a few new settings in your .env file (an API key, allowed dashboard address, capital per trade). Step 5 covers this.

Bottom line: merge the pull request → train models on Kaggle → if the evaluation says PASS, add the models to the repo → deploy → run in Paper mode for a while → only then consider Live.

Step 1 – Merge the pull request

  1. Open PR #12 (“Repair trading pipeline”) on GitHub.
  2. Review and merge it into main.

Nothing will start trading after the merge — the app waits for valid models.

Step 2 – Do we need to create the models again?

Short answer: yes, they must be trained again — but you don't have to build anything yourself. Everything is already written for you:

  • The old model files cannot be reused or “fixed”: they expect 5 numbers as input while the app produces 15, and the notebooks that made them are lost.
  • The repo now contains a ready-made training script (training/train_triad.py). Running it creates and trains all three models (Hunter, Guardian, Executive) in one go.
  • You only need to run it on Kaggle (free) and copy three files back. That's Step 3.

Step 3 – Train the models on Kaggle

  1. Go to kaggle.com → Create → New Notebook. CPU is fine.
  2. In the notebook settings, turn Internet ON.
  3. Cell 1 — get the code and install the libraries: 
    !git clone https://github.com/Premsh101/AITrader.git
%cd AITrader
!pip install -q "stable-baselines3==2.8.0" "numpy>=2.0" "pandas_ta==0.4.67b0" "yfinance>=0.2.40"
  4. Cell 2 — quick 2-minute test that everything runs (do this first!): 
    !python training/train_triad.py --smoke
!python training/evaluate_triad.py --smoke
    This also confirms all stock symbols still download correctly from Yahoo.
  5. Cell 3 — the real training (takes a few hours): 
    !python training/train_triad.py --timesteps 1000000
    Tip: if a session hits Kaggle's 12-hour limit, progress is saved every 100k steps in checkpoints/. You can also train one model per session with --brains hunter, then guardian, then executive.

Step 4 – Evaluate: the go / no-go gate

  1. Still on Kaggle, run: 
    !python training/evaluate_triad.py
  2. This replays the whole trading pipeline on data the models have never seen, including realistic trading costs, and prints a verdict:
    • VERDICT: PASS – the strategy beat simply holding the NIFTY index. OK to deploy.
    • VERDICT: FAIL – it did not. Do not deploy. Adjust the training settings and retrain. Never put a failing model in front of real money.
  3. On PASS: download these three files from Kaggle's file browser (inside AITrader/models/):
    • hunter_apex_1500_brain.zip
    • guardian_apex_1500_brain.zip
    • executive_apex_manager.zip
  4. Commit them into the repo's models/ folder (same names) and push.

Safety net: if a wrong or old file ends up in models/, the app detects the mismatch, refuses to trade, and shows an error in the dashboard.

Step 5 – Configure the server (.env)

  1. On your VPS, copy .env.example to .env.
  2. Generate a strong API key: 
    openssl rand -hex 32
  3. Fill in at minimum: 
    AITRADER_API_KEY=<the random key>
NEXT_PUBLIC_API_KEY=<the SAME key>
ALLOWED_ORIGINS=https://your-dashboard-domain.example
TRADE_CAPITAL_PER_SLOT=10000        # rupees per position
SHOONYA_USER=... (and the other broker fields, only needed for Live)

Why: every button that can change something (switch mode, place an order) now requires this key. Without it, strangers who find your server's address can do nothing.

Step 6 – Deploy with Docker

  1. Pull the latest main (with the new models) on the VPS.
  2. Run: 
    docker compose up --build -d
  3. Database tables are created automatically on first start (migrations run in the container entrypoint). Existing databases are upgraded in place — no manual steps.
  4. Important networking change: the app, dashboard and database now listen only on 127.0.0.1 (the server itself). Your reverse proxy (Coolify) must forward public traffic to 127.0.0.1:8005 (API) and 127.0.0.1:8006 (dashboard). If Coolify connects over a Docker network instead, adjust the ports: lines in docker-compose.yml.

Step 7 – Verify everything works

  1. Check the API: 
    curl http://127.0.0.1:8005/health
    Expect: {"status":"ok","brains_ready":true}. If brains_ready is false, the model files are missing or wrong — see Appendix B.
  2. Open the dashboard in your browser. The activity feed should show “Trading cycle started / complete” during market hours (Mon–Fri 09:15–15:30 IST). Outside those hours it stays quiet — that's normal.
  3. Try a manual paper order without the key — it must be rejected (401): 
    curl -X POST http://127.0.0.1:8005/order \
  -H 'Content-Type: application/json' \
  -d '{"symbol":"TCS","quantity":1,"side":"BUY","reference_price":4000}'

Step 8 – Run in Paper mode

  • The system starts in Paper mode: it simulates trades with real market prices but no real money and no broker connection.
  • Let it run for at least a few weeks. Watch the dashboard: total P&L, win rate, and whether its decisions look sane.
  • Paper results are recorded at the last daily closing price — treat them as an approximation, slightly optimistic versus reality.

Step 9 – Going Live (read carefully)

Live mode trades real money automatically. Only switch after: the evaluation PASSED, paper trading looked healthy for weeks, and broker credentials are set in .env.
  1. Flip the toggle in the dashboard header. You'll get a confirmation popup — this is deliberate; the backend also refuses the switch without it.
  2. Know the rules the system follows in Live mode:
    • Maximum 5 positions at once, about TRADE_CAPITAL_PER_SLOT rupees each.
    • Orders are intraday (product “I”): the broker force-closes anything still open around 15:20 IST. The app stops opening new positions after 15:15 IST for this reason.
    • Recorded fill prices are best-effort (quote just after the order), not exact fills — small differences vs. your broker statement are expected.
  3. To stop: flip back to Paper anytime, or docker compose down.

Appendix B – Troubleshooting

SymptomMeaningFix
Dashboard shows repeated ERROR “AI brains not ready … skipping trading cycle” The three model files are missing from models/, or they don't match the app (wrong shape). Do Steps 3–4, commit the three zips with the exact filenames, redeploy.
Button in dashboard does nothing / errors with 401 API key mismatch. Make sure AITRADER_API_KEY and NEXT_PUBLIC_API_KEY are identical, then rebuild: docker compose up --build -d (the dashboard bakes the key in at build time).
Mutating request returns 503 “API key not configured” AITRADER_API_KEY is empty on the server. Set it in .env and restart.
Switching to LIVE is rejected The extra safety check: the request must include confirm: true. Use the dashboard toggle (it sends it after your confirmation popup).
“No market data returned – skipping cycle” Yahoo Finance is unreachable or rate-limiting your server. Usually temporary. If persistent, raise FETCH_TTL_SECONDS; check the server's internet access.
Nothing happens all day It's outside NSE market hours (Mon–Fri 09:15–15:30 IST) — the loop sleeps by design. Wait for market hours, or set TRADING_ALWAYS_ON=1 temporarily to test.
A stock never trades / warnings about one symbol Its ticker may have been renamed or delisted on Yahoo. Remove or update it in app/stock_list.py.
